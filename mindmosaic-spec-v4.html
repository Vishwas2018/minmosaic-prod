1 System Overview

▶

MindMosaic is an online practice assessment platform for Australian students preparing for NAPLAN-style assessments in Years 3, 5, 7, and 9. It delivers fully online practice tests aligned with NAPLAN structure, domains, and evaluation criteria while operating strictly as a practice-only environment.

Domains

Writing, Reading, Conventions of Language (CoL), and Numeracy.

Users

RoleCapabilities
StudentTake practice exams, view results & feedback, track progress
TeacherView class reports, assign practice sets, review analytics
ParentView child's progress reports and domain strengths
AdminManage item bank, review attempts, invalidate exams, manage feature flags
GuestTake up to 3 practice exams without an account (no PII collected)

Scoring model

Automated scoring for objective questions; automated writing evaluation aligned with NAPLAN criteria (heuristic/LLM-based). All scores are practice estimates.

Label requirement: All student-visible scoring outputs must include: "Practice result — not an official NAPLAN score." MindMosaic must never claim parity with official NAPLAN marking.

Age & consent

  • Year 3 (~8), Year 5 (~10), Year 7 (~12), Year 9 (~14). All users under 16 require parental consent (Australian Privacy Act).
  • Guest mode collects no PII beyond a device fingerprint — no age gate required.
  • Writing submissions for users under 16 are subject to content safety scanning.

3 Non-Negotiable Constraints

▶

A. Confidentiality

  • Client must never receive answer keys — no correct answers, regex validators, numeric tolerances, or any derivable scoring data.
  • scoring_config never exposed to any frontend role — includes routing thresholds, band cutoffs, rubric mappings.
  • All API responses use role-specific DTOs returning minimum required fields.
  • During an attempt, students retrieve items only from the immutable snapshot, never live item bank tables.

B. Integrity

  • Server-authoritative timer: expires_at = started_at + time_limit_sec; client timers are display-only.
  • Immutable snapshots after attempt start (DB trigger enforced).
  • No client-side scoring — results read from persisted server output only.
  • Server-enforced state transitions via Postgres trigger (adjacency matrix).
  • Reproducibility: results reference exact snapshot_hash (SHA-256 canonicalized) + engine_version.
  • Server-side randomization during snapshot generation.
  • Max 1 non-terminal attempt per user at any time across all domains (prevents answer-farming).

C. Access Control

  • All tables: RLS deny-by-default.
  • All exam writes: Edge Functions only (service role).
  • Guest access: attempt_secret stored bcrypt-hashed; verified via HMAC challenge-response.
  • All identifiers: UUIDv4 (unguessable).
  • HSTS on all API responses (Strict-Transport-Security: max-age=31536000; includeSubDomains).

D. Abuse Prevention

  • Rate limiting on all endpoints (see §10).
  • 64 KB global request body limit on all Edge Functions, enforced before Zod parsing.
  • Per-field size limits in Zod schemas (text max 1000 chars, array fields max 50 entries).
  • Audit logging for all sensitive access.
  • Queue retry + dead-letter queue with alerting.
  • No sensitive data in client analytics/logs.

4 Exam Attempt Lifecycle (State Machine)

▶

States

StateDescriptionTerminal?
createdAttempt exists, timer not started, snapshots not lockedNo
startedstarted_at set, expires_at computed, snapshots lockedNo
in_progressAt least one persisted response exists (optional informational)No
submittedFinal submission accepted, no further writesNo

Allowed transitions

created → started
started → in_progress (optional)
started | in_progress → submitted
started | in_progress → expired
started | in_progress → abandoned
submitted → scored
expired → submitted (server-only auto-submit, auto_submitted=true)
any non-terminal → invalidated (admin/system only)

Enforcement

  • Postgres function validate_attempt_transition(current, new) → boolean encodes the adjacency matrix.
  • BEFORE UPDATE trigger on attempts; invalid transitions raise exception.
  • created → started must atomically: generate snapshots, set started_at/expires_at, store randomization_seed — single transaction.
  • Concurrent attempt guard: SELECT count(*) FROM attempts WHERE user_id=$1 AND status IN ('created','started','in_progress') FOR UPDATE. If ≥ 1, reject with CONCURRENT_ATTEMPT_LIMIT.

Expiry detection (dual-path)

  • Primary — check-on-read: Every Edge Function that reads/writes an attempt checks expires_at inline. If overdue, transitions to expired (or submitted if auto-submit enabled) within the same transaction. Sub-second detection.
  • Fallback — pg_cron: Runs every 60s with 5-second offset to avoid races: UPDATE attempts SET status='expired' WHERE status IN ('started','in_progress') AND expires_at < NOW() - INTERVAL '5 seconds'.

Guard rules

  • No submit after expiry unless within grace window (default 5s). Grace applies to submit only.
  • No writes in terminal states.
  • Scoring idempotent (submitted → scored).
  • Invalidation records invalidated_at, invalidated_by, invalidated_reason.
  • Start idempotent: repeated requests return existing attempt without resetting timer.

5 Timer & Expiry Enforcement

▶

Server timing

  • On created → started: started_at = NOW(), expires_at = started_at + time_limit_sec * INTERVAL '1 second'.
  • All expiry decisions use Postgres NOW() only.

Client sync

  • API returns { server_now, expires_at }. Client computes drift and adjusts display.
  • Client timer is cosmetic — prompts submit at zero but does not lock UI.
  • Client polls GET /attempt/:id/status every 60s to detect server-side expiry.

Submit finalization (transactional)

  1. Validate status is started/in_progress.
  2. Validate NOW() ≤ expires_at + grace_window.
  3. Atomically: persist final responses, set submitted_at, status → submitted.
  4. Enqueue scoring job (async — does NOT block).
  5. Return immediately: { status: 'submitted' }.

Submit idempotency

Via submission_id (client-generated UUID). Repeated submit with same ID returns existing outcome.

Autosave — atomic stale-write protection

Monotonic client_revision per item. Enforcement is in the SQL upsert, not application code:

INSERT INTO attempt_responses
  (attempt_id, item_snapshot_id, response_payload, client_revision, saved_at)
VALUES ($1, $2, $3, $4, now())
ON CONFLICT (attempt_id, item_snapshot_id)
DO UPDATE SET
  response_payload = EXCLUDED.response_payload,
  client_revision  = EXCLUDED.client_revision,
  saved_at         = now()
WHERE attempt_responses.client_revision < EXCLUDED.client_revision
RETURNING id;

If zero rows returned → STALE_REVISION (409, retryable). SELECT ... FOR UPDATE on the attempts row serializes per-attempt writes.

6 Anti-Cheat Controls

▶

Trust model

Client events are untrusted telemetry. Only the server decides warnings, auto-submit, or invalidation. Enforcement is deterministic: same event stream → same outcome.

Event contract

{ attempt_id, event_type, client_event_seq, client_event_ts_ms, observed_state }

event_type ∈ { fullscreen_exit, focus_loss, context_menu }

Server rules

  • Accept only if status is started/in_progress.
  • Sequence guard: ignore events where client_event_seq ≤ last_processed_event_seq.
  • Dedup window: don't increment counters within DEDUP_WINDOW_MS.

Thresholds

ViolationAction
Fullscreen exit1–2 → warning; 3 → auto-submit
Focus loss1–3 → warning; 4 → auto-submit
Right-clickDeterrence only; log + correlate

Enforcement

  • Warnings: persist last_warning_type, last_warning_at; return warning to client.
  • Auto-submit: atomically set auto_submitted=true, reason, status → submitted, lock writes.
  • Idempotent: repeated triggers return existing submitted state.
  • Counter + threshold + transition: single atomic transaction.

7 Writing Module

▶

Editor constraints

  • Plain text only — no rich text, no formatting tools.
  • Paste disabled (onPaste intercepted).
  • Spellcheck disabled (spellcheck=false, autocorrect=off).
  • Autosave every 5s (debounced from last keystroke).
  • Hard character limit: 3,000 chars at textarea level + server enforcement.

Word limits

Min: 20 words (submit-time). Max: 350 words (save + submit).

Deterministic word count

function countWords(text: string): number {
  return text
    .replace(/\n/g, ' ')
    .replace(/[^\w\s]/g, '')
    .replace(/\s+/g, ' ')
    .trim()
    .split(' ')
    .filter(t => t.length > 0)
    .length;
}

Both client and server implement identical logic; server is authoritative.

Prompt display

  • Stimulus + task instruction displayed above editor, read-only, from snapshot (not live data).
  • May include images (signed Supabase Storage URLs, 2-hour expiry).

8 Content Selection & Snapshots

▶

Selection rules

  • Questions selected from published pools per (domain, year_level).
  • No-repeat window: last 50 attempts. If pool exhausted: allow repeats, flag attempt.
  • Writing prompts: no-repeat window of 50 per year_level.

Snapshot rules

  • Question list, order, option order, and prompt all frozen at attempt start (immutable).
  • Snapshot records are append-only — BEFORE UPDATE trigger raises exception.
  • Randomization seed stored with snapshot for reproducibility.

Snapshot hash specification

snapshot_hash = SHA-256 of canonicalized JSON (keys sorted recursively, no whitespace).

// packages/shared/src/hash.ts
import { createHash } from 'crypto';

export function computeSnapshotHash(data: unknown): string {
  const canonical = JSON.stringify(data, (_, v) =>
    v && typeof v === 'object' && !Array.isArray(v)
      ? Object.keys(v).sort().reduce((o, k) => { o[k] = v[k]; return o; }, {} as any)
      : v
  );
  return createHash('sha256').update(canonical).digest('hex');
}

9 Guest Mode

▶

Caps

3 attempts lifetime (best-effort) via guest_device_id. Secondary: rate limit by IP/User-Agent.

Security

  • attempt_secret stored bcrypt-hashed server-side.
  • Plaintext secret returned once only in createGuestAttempt response. Never logged.
  • Guest writes via Edge Functions verifying secret.
  • Session tokens: 15-min, bound to attempt_id, non-renewable.

Transport security

  • HSTS on all responses.
  • Challenge-response verification: client sends HMAC-SHA256(attempt_id + ":" + timestamp, secret). Timestamp must be within ±60s of server time.
  • Brute-force protection: after 3 consecutive failures for same attempt_id, exponential backoff (5s → 30s → 10min block). Tracked via guest_verify_failures counter.

12 Scoring & Results Pipeline

▶
Key design: Scoring is always async. submitAttempt enqueues a job and returns immediately. The client polls or subscribes via Realtime until status = scored.

Scoring job queue

scoring_jobs (
  id              uuid PK,
  attempt_id      uuid NOT NULL REFERENCES attempts,
  idempotency_key uuid NOT NULL UNIQUE, -- same as submission_id
  job_type        text CHECK (IN ('objective','writing')),
  status          text CHECK (IN ('pending','processing','completed','failed','dead')),
  retries / max_retries / last_error / timestamps
)

Insert uses ON CONFLICT (idempotency_key) DO NOTHING — duplicate submits never create duplicate jobs.

Worker flow

  1. Pick oldest pending: UPDATE ... WHERE id = (SELECT id ... FOR UPDATE SKIP LOCKED) RETURNING *
  2. If attempt already scored → mark job completed, exit.
  3. Load responses + item_scoring_config.
  4. Evaluate: mcq=exact match, multi_select=set compare, short_text=normalised match, numeric=tolerance, drag_drop/dropdown/order=exact match.
  5. Persist attempt_item_results + attempt_results.
  6. Transition attempt → scored.
  7. On failure: retry up to max_retries, then move to dead (DLQ alert fires).

Writing scoring

Separate job_type='writing' job. Evaluates against 10 NAPLAN criteria. Persists to writing_results. Content safety scan in parallel. Does not block objective scoring.

Results display

  • Client shows "Scoring your exam…" between submitted and scored.
  • All results include practice disclaimer.
  • Writing may show "Scoring in progress…" independently.

15 Scalability & Ingestion Pipeline

▶

Load assumptions

  • 5,000 students start exams within 10 minutes.
  • Autosave every 5s → ~1,000 writes/sec peak.
  • Payload: objective ~1 KB, writing ~2–4 KB.

Async ingestion (event-driven)

  • Edge Functions insert into ingest_events (append-only, cheap writes).
  • Primary: AFTER INSERT trigger calls pg_notify('ingest_channel', NEW.id). Persistent worker listens and processes immediately (sub-second latency).
  • Alternative: Supabase Realtime subscription on ingest_events (1–2s latency).
  • Fallback: pg_cron every 60s catches anything missed during worker restart.
  • Worker drains to canonical tables with idempotency + atomic revision enforcement.

Worker responsibilities

  • Validate attempt active + not expired (check-on-read).
  • Validate item belongs to snapshot.
  • Enforce monotonic revision (atomic SQL).
  • Compute server-side word count; enforce max.
  • Mark processed; failures → dead_letter.

Backpressure

If backlog > 10,000 rows: switch to batch mode (500/cycle), emit alert. Rate limiting and sampling available as emergency measures.

SQL function for atomic upsert

CREATE FUNCTION upsert_response(
  p_attempt_id uuid,
  p_item_snapshot_id uuid,
  p_response_payload jsonb,
  p_client_revision int
) RETURNS TABLE(rows_affected int) AS $$
  WITH upserted AS (
    INSERT INTO attempt_responses
      (attempt_id, item_snapshot_id, response_payload, client_revision, saved_at)
    VALUES (p_attempt_id, p_item_snapshot_id, p_response_payload, p_client_revision, now())
    ON CONFLICT (attempt_id, item_snapshot_id)
    DO UPDATE SET
      response_payload = EXCLUDED.response_payload,
      client_revision  = EXCLUDED.client_revision,
      saved_at         = now()
    WHERE attempt_responses.client_revision < EXCLUDED.client_revision
    RETURNING 1
  )
  SELECT count(*)::int AS rows_affected FROM upserted;
$$ LANGUAGE sql;

// Called via pg_notify listener, cron, or HTTP trigger

async function processNextScoringJob(db: SupabaseClient) {
  // 1. Pick job (FOR UPDATE SKIP LOCKED)
  const { data: job } = await db.rpc('pick_scoring_job');
  if (!job) return; // No pending jobs

  try {
    // 2. Load attempt
    const { data: attempt } = await db.from('attempts')
      .select('*').eq('id', job.attempt_id).single();

    // Already scored? Mark job done.
    if (attempt.status === 'scored') {
      await db.from('scoring_jobs').update({
        status: 'completed', completed_at: new Date().toISOString()
      }).eq('id', job.id);
      return;
    }

    // 3. Load responses
    const { data: responses } = await db.from('attempt_responses')
      .select('*').eq('attempt_id', job.attempt_id);

    // 4. Load snapshot (to get source_item_ids)
    const { data: snapshot } = await db.from('attempt_snapshots')
      .select('snapshot_data').eq('attempt_id', job.attempt_id).single();

    // 5. Load scoring configs for all source items
    const sourceItemIds = snapshot.snapshot_data.map(
      (s: any) => s.source_item_id
    );
    const { data: configs } = await db.from('item_scoring_config')
      .select('*').in('item_id', sourceItemIds);

    // 6. SCORE EACH ITEM
    const itemResults = [];
    let correctCount = 0;

    for (const snapItem of snapshot.snapshot_data) {
      const response = responses.find(
        (r: any) => r.item_snapshot_id === snapItem.item_snapshot_id
      );
      const config = configs.find(
        (c: any) => c.item_id === snapItem.source_item_id
      );

      const isCorrect = evaluateResponse(
        snapItem.question_type,
        response?.response_payload,
        config
      );

      itemResults.push({
        attempt_id: job.attempt_id,
        item_snapshot_id: snapItem.item_snapshot_id,
        is_correct: isCorrect,
        score_awarded: isCorrect ? config.max_score : 0,
        max_score: config.max_score,
      });

      if (isCorrect) correctCount++;
    }

    // 7. PERSIST RESULTS
    await db.from('attempt_item_results').insert(itemResults);

    const totalItems = snapshot.snapshot_data.length;
    await db.from('attempt_results').insert({
      attempt_id: job.attempt_id,
      domain: attempt.domain,
      year_level: attempt.year_level,
      total_items: totalItems,
      correct_count: correctCount,
      score_raw: correctCount,
      score_pct: totalItems > 0 ? (correctCount / totalItems) * 100 : 0,
      band_estimate: computeBand(correctCount, totalItems),
      engine_version: ENGINE_VERSION,
    });

    // 8. TRANSITION → scored
    await db.from('attempts').update({
      status: 'scored',
      scored_at: new Date().toISOString(),
    }).eq('id', job.attempt_id);

    // 9. MARK JOB COMPLETED
    await db.from('scoring_jobs').update({
      status: 'completed', completed_at: new Date().toISOString()
    }).eq('id', job.id);

  } catch (error) {
    // Retry or dead-letter
    const newRetries = job.retries + 1;
    if (newRetries >= job.max_retries) {
      await db.from('scoring_jobs').update({
        status: 'dead', last_error: error.message
      }).eq('id', job.id);
      await db.from('dead_letter_queue').insert({
        source_table: 'scoring_jobs', source_id: job.id,
        error: error.message, payload: job,
      });
    } else {
      await db.from('scoring_jobs').update({
        status: 'pending', retries: newRetries,
        last_error: error.message, started_at: null,
      }).eq('id', job.id);
    }
  }
}

function evaluateResponse(
  questionType: string,
  responsePayload: any,
  config: any
): boolean {
  if (!responsePayload || !config) return false;

  switch (questionType) {
    case 'mcq':
      return responsePayload.selected_option_id ===
        config.correct_response.correct_option_id;
    case 'short_text':
      return normalizeText(responsePayload.text) ===
        normalizeText(config.correct_response.text);
    default:
      return false; // Other types in later phases
  }
}

function normalizeText(text: string): string {
  return text.toLowerCase().trim().replace(/\s+/g, ' ');
}

// stores/examStore.ts
import { create } from 'zustand';

interface ExamState {
  attempt: Attempt | null;
  snapshot: SnapshotItem[];
  responses: Map<string, { payload: any; revision: number }>;
  currentIndex: number;
  timerRemainingSec: number;
  serverDriftMs: number;
  flaggedItems: Set<string>;

  setAttempt: (attempt: Attempt, snapshot: SnapshotItem[], serverNow: string) => void;
  setResponse: (itemSnapshotId: string, payload: any) => void;
  setCurrentIndex: (index: number) => void;
  toggleFlag: (itemSnapshotId: string) => void;
  decrementTimer: () => void;
}

export const useExamStore = create<ExamState>((set, get) => ({
  attempt: null,
  snapshot: [],
  responses: new Map(),
  currentIndex: 0,
  timerRemainingSec: 0,
  serverDriftMs: 0,
  flaggedItems: new Set(),

  setAttempt: (attempt, snapshot, serverNow) => {
    const drift = new Date(serverNow).getTime() - Date.now();
    const remaining = Math.max(0,
      Math.floor((new Date(attempt.expires_at!).getTime() - Date.now() - drift) / 1000)
    );
    set({ attempt, snapshot, serverDriftMs: drift, timerRemainingSec: remaining });
  },

  setResponse: (itemSnapshotId, payload) => {
    const responses = new Map(get().responses);
    const existing = responses.get(itemSnapshotId);
    const revision = (existing?.revision ?? 0) + 1;
    responses.set(itemSnapshotId, { payload, revision });
    set({ responses });
  },

  setCurrentIndex: (index) => set({ currentIndex: index }),
  toggleFlag: (id) => {
    const flags = new Set(get().flaggedItems);
    flags.has(id) ? flags.delete(id) : flags.add(id);
    set({ flaggedItems: flags });
  },
  decrementTimer: () => set((s) => ({
    timerRemainingSec: Math.max(0, s.timerRemainingSec - 1)
  })),
}));

// hooks/useAutosave.ts
export function useAutosave() {
  const { attempt, responses } = useExamStore();
  const saveResponse = useSaveResponse();
  const pendingRef = useRef<Map<string, { payload: any; revision: number }>>(new Map());
  const savingRef = useRef(false);

  // Track dirty responses
  useEffect(() => {
    const unsub = useExamStore.subscribe((state, prev) => {
      for (const [id, resp] of state.responses) {
        const prev_resp = prev.responses.get(id);
        if (!prev_resp || prev_resp.revision !== resp.revision) {
          pendingRef.current.set(id, resp);
        }
      }
    });
    return unsub;
  }, []);

  // Flush every 5 seconds
  useEffect(() => {
    const interval = setInterval(async () => {
      if (savingRef.current || pendingRef.current.size === 0) return;
      savingRef.current = true;

      for (const [itemId, { payload, revision }] of pendingRef.current) {
        try {
          await saveResponse.mutateAsync({
            attempt_id: attempt!.id,
            item_snapshot_id: itemId,
            response_payload: payload,
            client_revision: revision,
          });
          pendingRef.current.delete(itemId);
        } catch (err: any) {
          if (err.code === 'STALE_REVISION') {
            pendingRef.current.delete(itemId); // Will re-queue on next change
          } else if (err.code === 'ATTEMPT_EXPIRED') {
            // Navigate to expired screen
            navigateToExpired();
          }
          // Other errors: keep in pending for next cycle
        }
      }
      savingRef.current = false;
    }, AUTOSAVE_INTERVAL_MS);

    return () => clearInterval(interval);
  }, [attempt?.id]);
}
